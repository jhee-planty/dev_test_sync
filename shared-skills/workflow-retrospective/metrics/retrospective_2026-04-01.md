## 비효율 분석 (5차원)

### 1. 소요 시간

- **폴링 대기가 전체의 60%+**: #113/#114 결과 대기에만 30분+ 순수 폴링 (45초 × ~40회).
  이 시간 동안 dev PC는 아무 작업도 하지 못함.
- **초기 스캔은 효율적**: #085~#094 (10건/50분)은 5분/건으로 매우 빠른 사이클.
- **빌드-배포-테스트 사이클 개선**: B7 빌드 전체 (코드 수정~결과 수신)이 약 30분. 이전 ~22분/빌드에서 약간 증가했으나, 2개 서비스 동시 변경이므로 실질 효율은 향상.
- **#095~#103 구간 5.2시간**: 9건 테스트에 5시간은 과도. 이 구간에 context break + test PC 대기가 포함.

### 2. 실패 패턴

**구조적 실패 패턴 (신규 발견)**:

| 패턴 | 해당 서비스 | 근본 원인 | 해결 가능성 |
|------|-----------|-----------|------------|
| SPA fetch/XHR 에러 핸들러 | Grok, GitHub Copilot, Gamma | 프론트엔드가 모든 비정상 응답을 catch | 응답 포맷으로는 불가 |
| H2 멀티플렉싱 충돌 | Gemini | 서버 응답과 블록 응답 충돌 or GOAWAY 페이지 리셋 | is_http2 어떤 값도 불가 |
| Record-based sync | Notion | fake UUID → syncRecordValue timeout | NDJSON으로는 불가 |

**핵심 인사이트**: 이 3가지 패턴은 모두 **"APF가 응답을 교체하는 방식(escalation ①)의 구조적 한계"**를 보여준다.
SPA 기반 서비스에서 API 응답 교체로 경고를 전달하는 것은 원천적으로 불가능할 수 있다.
escalation ②③(에러 페이지, JS injection)이 필요하나 APF에 응답 수정 기능이 없어 구현 불가.

### 3. 불필요한 동작

- **Grok SSE 시도 (#111)**: Design doc에 "REST JSON (Phase 1 보고) — 코드는 SSE"라고 되어 있었으나,
  HAR 캡처 없이 SSE를 시도. HAR 캡처(#113)를 먼저 하고 포맷을 확인했어야 함.
  → **HAR 캡처 선행 규칙 필요**
- **Gemini GOAWAY=false (#112)**: 이전 실패(is_http2=2→protocol error, is_http2=1+GOAWAY→page reset)
  패턴으로 볼 때 GOAWAY만 바꿔도 동일 결과가 예측 가능했음. 그러나 "시도해보지 않으면 모른다"는 논리로 빌드.
  → 빌드 1회 소모. 예측 가능한 실패에 빌드를 쓰지 않는 규칙 필요.
- **폴링 과다 반복**: 45초 간격 × 40회+ = 30분+ 순수 폴링. test PC가 비활성이면 무한 폴링.
  **폴링 타임아웃**(예: 15분) 설정 필요.

### 4. 워크플로우 병목

- **test PC 가용성 불확실**: #113/#114 요청 후 30분+ 결과 미도착. test PC가 동작 중인지 확인할 방법이 etap 로그뿐.
- **폴링 모드의 생산성 제로**: 폴링 중 다른 작업을 하지 않는 규칙(cowork-remote § 2-1)이 있어
  결과 대기 동안 dev PC는 완전히 유휴. 이 규칙의 의도(결과 미수신 시 무관한 작업 방지)는 이해하지만,
  **분석 작업이나 다음 서비스 설계**는 허용해도 될 것.
- **context break 복구**: 이 세션에서도 context break 후 상태 재구성에 시간 소모.

### 5. 자원 활용

- **컴파일 서버**: 안정적. ninja 빌드 1분 이내. 문제 없음.
- **테스트 서버**: etapd 정상 동작. 배포 검증 게이트 적용으로 미배포 테스트 방지됨.

## 신규 개선안

### [CRITICAL] 개선안 #1: 폴링 타임아웃 + 비동기 전환

- **문제**: #113/#114 대기에 30분+ 순수 폴링. dev PC 완전 유휴.
- **근거**: 45초 × 40회+ = 30분. 이 시간 동안 생산적 작업 제로.
- **제안**:
  ① 폴링 최대 15분(20회). 초과 시 "결과 미도착, test PC 상태 확인 필요" 보고 후 다른 작업 진행.
  ② cowork-remote § 2-1 "폴링 중 다른 동작 금지" 규칙을 **완화**: 코드 수정/빌드는 금지하되,
     다음 서비스 설계, status 갱신, impl journal 정리 등 **분석 작업은 허용**.
  ③ 결과 미도착 시 "다음 서비스로 이동, 결과 도착 시 복귀" 패턴 도입.
- **대상**: `cowork-remote/SKILL.md` § 2-1, `genai-warning-pipeline/SKILL.md`
- **기대**: 유휴 시간 30분+ → 0. 폴링 대기를 생산적 시간으로 전환.
- **우선순위**: CRITICAL (전체 시간의 60%가 유휴)

### [HIGH] 개선안 #2: HAR 캡처 선행 규칙

- **문제**: Grok SSE 시도(#111)가 실패. 실제 응답 포맷을 모른 채 generator 작성.
- **근거**: Design doc에 "REST JSON (Phase 1 보고) — 코드는 SSE"라는 모호한 정보.
  HAR 캡처(#113) 없이 SSE를 시도하여 1빌드 낭비.
- **제안**: **새 서비스의 generator 작성 전 HAR 캡처 필수**. Design doc의 comm_type이
  불확실하거나 "추정"인 경우, Phase 3 진입 전 HAR 캡처를 Phase 2에 편입.
  ```
  Phase 2 design checklist 추가:
  - [ ] HAR 캡처로 실제 응답 Content-Type 확인
  - [ ] SSE/JSON/NDJSON/기타 구분 확인
  - [ ] 응답 body 구조 최소 1건 샘플 확보
  ```
- **대상**: `apf-warning-design/SKILL.md`, `apf-warning-design/references/warning-delivery-checklist.md`
- **기대**: 추정 기반 빌드 방지. 1~2빌드/서비스 절약.
- **우선순위**: HIGH

### [HIGH] 개선안 #3: 예측 가능한 실패에 빌드 쓰지 않기

- **문제**: Gemini GOAWAY=false(#112)는 기존 패턴(is_http2=2→protocol error, is_http2=1+GOAWAY→reset)으로
  "GOAWAY만 바꿔도 page reset"이 예측 가능했음.
- **근거**: is_http2=1은 서버 연결을 끊으므로 H2 멀티플렉싱이 깨짐. GOAWAY 유무와 무관하게
  다른 스트림이 영향 받음. 이 분석을 빌드 전에 수행했으면 1빌드 절약.
- **제안**: 빌드 전 **"예측 분석" 단계** 추가:
  ```
  변경 전 체크:
  1. 이전 실패 패턴과 이번 변경이 다른 점은?
  2. 이번 변경으로 이전 실패의 근본 원인이 해결되는가?
  3. "아니오"라면 → 빌드하지 않고 다른 접근법 탐색
  ```
- **대상**: `apf-warning-impl/SKILL.md` — Step 2
- **기대**: 예측 가능한 실패 빌드 방지. 1빌드/서비스 절약.
- **우선순위**: HIGH
