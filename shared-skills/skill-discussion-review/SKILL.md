# Skill Discussion Review

## Purpose

기존 스킬을 **구조화된 다자간 토론**으로 점검하여 개선안을 도출한다.
단순한 체크리스트 리뷰가 아니라, 서로 다른 전문 관점에서 충분히 논의한 뒤
합의를 통해 수정 사항을 결정한다.

**왜 토론인가:**
스킬 개선은 단일 관점으로 보면 놓치는 부분이 많다. 예를 들어 자동화 엔지니어는
실행 안정성에 집중하지만, 보안 전문가는 인증 문제를 먼저 본다.
토론은 이런 관점 충돌을 표면화하고, 실제 환경에서 발생할 문제를 사전에 잡는다.
이번 방식이 효과적이었던 이유는 "반론 → 재반론 → 절충"을 통해
단일 관점의 맹점이 보완되었기 때문이다.

---

## Participants — 고정 + 자동 구성

### 필수 참여자 (항상 포함)

| 역할 | 약칭 | 핵심 기능 |
|------|------|----------|
| **토론 전문가** (Discussion Facilitator) | DF | 논점 구조화, 라운드 진행, 합의 도출. 성급한 결론 방지. |
| **외부 컨설턴트** (External Consultant) | EC | 도메인 외부 시각 제공. "왜 이렇게 하는가?"를 근본적으로 질문. |

### 자동 구성 참여자 (대상 스킬에 따라 선정)

대상 스킬의 성격을 분석하여 3~4명의 추가 참여자를 자동 선정한다.

→ See `references/participant-roles.md` for 역할 풀 및 선정 기준.

**선정 원칙:**
1. 대상 스킬의 **주요 도메인**에서 최소 2명 (예: 네트워크 + 자동화)
2. 대상 스킬의 **인접 도메인**에서 1명 (예: 보안, UX, 운영)
3. 역할 간 **관점 충돌**이 발생하도록 구성 (모두 동의하면 토론이 아님)

**예시:**
- test-pc-worker 점검 → TA(테스트 자동화) + IS(인프라/보안) + PA(파이프라인) + SA(스킬 문서)
- apf-warning-design 점검 → FE(프론트엔드) + BE(백엔드) + UX(사용자 경험) + SA(스킬 문서)
- etap-build-deploy 점검 → DevOps(빌드/배포) + QA(품질) + IS(인프라) + SA(스킬 문서)

## Discussion Protocol

### Phase 1 — 사전 분석 (토론 전)

토론 시작 전에 대상 스킬을 읽고 분석한다. 토론 없이 이 단계만 진행.

```
1. 대상 스킬의 SKILL.md + references/ 전체 읽기
2. 관련 스킬이 있으면 함께 읽기 (연동/의존 관계)
3. 최근 실제 사용 이력 확인 (테스트 결과, 실패 로그 등)
4. 참여자 자동 구성 — 사용자에게 구성 결과를 보여주고 확인받기
5. 논점 목록 초안 작성 (각 참여자 관점에서 1~2개씩)
```

사용자가 특정 문제를 언급했으면 해당 문제를 논점 목록 상위에 배치.
사용자가 "전반적으로 점검"이라고 했으면 Phase 1에서 발견한 이슈 기반으로 구성.

### Phase 2 — 구조화 토론

→ See `references/discussion-protocol.md` for 라운드 진행 상세 규칙.

**핵심 원칙:**

1. **성급한 합의 금지** — DF가 "합의가 형성되고 있다"고 판단해도, 반드시
   반론 기회를 제공한 후에만 합의를 선언한다.
2. **모든 참여자 발언 보장** — 각 라운드에서 최소 3명 이상이 실질적 의견을 제시.
   "동의합니다"만으로는 충분하지 않다 — 이유나 보완 의견이 있어야 한다.
3. **반론 의무** — EC(외부 컨설턴트)는 매 라운드에서 최소 1회 "그러면 ~는 어떻게 되나?"
   형식의 반론 또는 의문을 제기해야 한다.
4. **실증 기반** — 주장에는 근거(코드, 로그, 테스트 결과, 경험)를 제시.
   "~할 것 같다"보다 "실제로 ~가 발생했다"가 우선.
5. **라운드 구조** — 하나의 논점에 대해 최소 2라운드 이상 진행.
   단, 참여자 전원이 동일한 결론에 도달한 명백한 사안은 1라운드로 축약 가능.

**라운드 진행:**

```
Round N — [논점 제목]

DF: 논점 제시 + 배경 설명 + 각 참여자에게 질문

[참여자 1]: 관점 제시 (근거 포함)
[참여자 2]: 다른 관점 또는 보완
EC: 반론 또는 근본적 질문
[참여자 3]: EC 반론에 대한 응답

DF: 중간 정리 + 추가 논의 필요 여부 판단
  → 추가 필요 → Round N 계속
  → 합의 형성 → 합의 사항 명시 후 다음 논점으로
```

### Phase 3 — 합의 정리 및 수정 사항 산출

토론 종료 후, DF가 최종 합의를 정리한다.

```
## 최종 합의

### 합의 사항 1: [제목]
- 내용: ...
- 근거: ...
- 반대 의견 (있었다면): ...
- 수정 대상 파일: ...

### 합의 사항 2: [제목]
...

### 미합의 사항 (있다면)
- 내용: ...
- 대립 관점: A vs B
- 권장 후속 조치: ...
```

**수정 사항 목록:**
합의에서 도출된 구체적 파일 수정 내역을 표 형태로 정리.

| # | 파일 | 수정 내용 | 근거 라운드 |
|---|------|----------|-----------|
| 1 | SKILL.md § Step 4 | git_sync.sh → git_sync.bat | Round 3 |
| 2 | git-push-guide.md | 전면 재작성 | Round 4 |

이 표가 사용자의 "수정 진행해줘" 지시 시 실행 목록이 된다.

### Phase 4 — 검증 계획 (선택)

수정 적용 전에 검증이 필요한 사항이 있으면 검증 계획을 제시.
검증이 불필요한 경우 (문서 수정만 등) 이 단계를 건너뛴다.

## Triggering Context

이 스킬은 다음 상황에서 트리거된다:

**직접 요청:**
- "이 스킬을 토론으로 점검해줘"
- "토론을 진행해줘"
- "다각도로 리뷰해줘"

**간접 신호:**
- "성급하게 결론 내리지 말고" → 사용자가 깊은 논의를 원함
- "모든 참가자의 의견을 충분히 고려해줘" → 다자간 토론 필요
- "왜 이 방식이어야 하는지 논의해줘" → 단순 리뷰가 아닌 토론 필요

**사용자가 문제를 제시한 경우:**
문제 보고서나 실패 사례가 주어지면, 해당 내용을 논점 목록의 핵심으로 삼아
토론을 구성한다. (이번 세션의 "4가지 실제 문제" 토론이 이 패턴)

## Integration with Other Skills

**skill-review-deploy와의 관계:**
- skill-review-deploy: 자동화된 품질 체크리스트 기반 리뷰 + 배포
- skill-discussion-review (이 스킬): 다자간 토론 기반 심층 점검
- 둘은 보완 관계. 이 스킬로 토론 → 수정 사항 도출 후,
  skill-review-deploy로 패키지 + 배포하는 순서가 자연스럽다.

**skill-creator와의 관계:**
- skill-creator: 새 스킬 생성 및 eval 기반 개선
- skill-discussion-review: 기존 스킬의 토론 기반 점검
- 새 스킬 생성 후 이 스킬로 점검하는 것도 가능하다.
