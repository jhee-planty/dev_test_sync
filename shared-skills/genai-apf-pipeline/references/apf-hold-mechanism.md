# APF H2 Hold 메커니즘 — 핵심 아키텍처

## 개요

APF(ai_prompt_filter)는 HTTP/2 POST 요청의 body를 중간에서 **hold(붙잡기)**하여
키워드 검사를 수행한다. 이 문서는 hold 메커니즘의 동작 원리와 **과차단 방지 원칙**을
설명한다.

---

## Hold 동작 흐름

```
클라이언트(브라우저)           Etap(MITM 브릿지)              원본 서버(AI 서비스)
       │                           │                              │
       │── H2 HEADERS (POST) ────→ │                              │
       │── H2 DATA (frame 1) ────→ │ ← body 버퍼에 축적 (hold)    │
       │── H2 DATA (frame 2) ────→ │ ← body 버퍼에 축적           │
       │── H2 DATA (END_STREAM) ──→│ ← body 완전 수신             │
       │                           │                              │
       │                     ┌─────┴─────┐                        │
       │                     │ 키워드 검사 │                        │
       │                     └─────┬─────┘                        │
       │                           │                              │
       │               ┌───── 결과 분기 ─────┐                    │
       │               │                     │                    │
       │          키워드 없음              키워드 발견              │
       │               │                     │                    │
       │         hold_release()         block_session()           │
       │               │                     │                    │
       │               │──── 버퍼 전달 ──────→│                    │
       │               │                     │                    │
       │               │              ← 차단 응답 생성             │
       │←──────────────│── 차단 응답 ──│                           │
       │               │  (원본에 미전달)                          │
```

---

## 왜 Hold가 필요한가

HTTP/2 body는 여러 DATA 프레임에 나눠서 전송될 수 있다.

```
예시: "내 주민번호는 990101-1234567이야" 라는 프롬프트가 2개 프레임으로 분할:
  Frame 1: "내 주민번호는 990101-"     ← 이것만 보면 키워드 없음
  Frame 2: "1234567이야"              ← 이것도 단독으로는 불완전
  전체 body: "내 주민번호는 990101-1234567이야"  ← 합쳐야 SSN 패턴 매칭
```

첫 프레임만 보고 "깨끗하다"고 판단하여 서버로 보내면, 키워드가 포함된 전체 메시지가
AI 서비스에 그대로 도달한다. **전체 body를 다 받은 후 한 번에 검사**해야 하므로
hold가 필수이다.

---

## 프레임 경계 버퍼링 (Boundary Buffer)

프레임 경계에서 키워드가 분할될 수 있으므로, APF는 **마지막 N바이트를 별도 버퍼에
저장**하고, 새로 수신된 데이터와 결합하여 검사한다.

```
이전 프레임 끝 (버퍼): "...990101-"
새 프레임 시작:        "1234567이야..."
결합 검사 대상:        "990101-1234567이야"  → SSN 패턴 매칭!
```

이 로직이 없으면 프레임 경계에 걸친 키워드를 놓친다.

---

## 과차단 방지 원칙 (Critical Rule)

> **민감정보가 포함된 패킷 이외에는 절대로 패킷 전송을 방해하는 동작이 있으면 안 된다.**

이것은 APF의 최우선 원칙이다. 위반 시 서비스 장애로 이어진다.

### 원칙 적용

| 상황 | 올바른 동작 | 잘못된 동작 |
|------|-----------|-----------|
| body에 키워드 있음 | block_session() → 차단 응답 | ✅ |
| body에 키워드 없음 | hold_release() → 즉시 서버로 전달 | ❌ 지연, 변조, 누락 |
| 비 API 트래픽 (로그인, 텔레메트리 등) | 검사 후 통과 | ❌ 차단하거나 지연 |
| 키워드가 AI 프롬프트가 아닌 곳에 존재 | 과차단 — 키워드/도메인 패턴 수정 필요 | ❌ 그대로 차단 |

### DB 설정과의 관계

| 컬럼 | 값 | 의미 |
|------|---|------|
| `h2_hold_request` | 0 | hold 안 함 — 프레임 즉시 전달 (h2_mode=1 서비스) |
| `h2_hold_request` | 1 | hold 함 — 전체 body 수신 후 검사 (h2_mode=2 서비스) |

**패턴**: `h2_mode=2` (연결 유지형) 서비스는 거의 항상 `h2_hold_request=1`.
`h2_mode=1` (연결 종료형)은 연결 자체가 끊기므로 hold 불필요.

## 과차단 사례: wrtn 로그인 (2026-04-17)

### 문제

wrtn.ai의 로그인 요청 body가 키워드 검사에 걸려 차단됨:

```json
{"email":"user@example.com","password":"myPassword123"}
```

`password` 키워드가 AC(Aho-Corasick) 패턴에 등록되어 있어,
로그인 API body의 JSON 키 이름 `"password"` 에 매칭되었다.

### 원인

- 도메인 패턴 `wrtn.ai,*.wrtn.ai`가 인증 서버까지 포함
- 키워드 `password`가 JSON 키 이름에도 매칭 (값뿐 아니라 키에도)
- hold → block_session 경로로 처리됨 (hold_release가 아님)

### 진단 방법

ai_prompt 로그에서 차단된 request body를 확인하여 과차단 여부를 판별:

```bash
ssh -p 12222 solution@218.232.120.58 \
  "grep -a '\"wrtn\"' /var/log/ai_prompt/$(date +%Y-%m-%d).log | grep password | tail -5"
```

etap.log의 `block_session`만으로는 차단 사실만 알 수 있고,
body 내용(과차단 여부)은 ai_prompt 로그에서만 확인 가능하다.

### 해결 방향

1. **경로 패턴 정밀화**: 인증 엔드포인트(`/auth/login` 등)를 검사 대상에서 제외
2. **키워드 컨텍스트 검사**: JSON 키 이름이 아닌 값에서만 매칭하도록 (인프라 확장 필요)
3. **도메인 분리**: 인증 도메인(`auth.wrtn.ai`)과 AI API 도메인(`api.wrtn.ai`)을 분리하여 등록
