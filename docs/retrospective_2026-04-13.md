## 비효율 분석

### 패턴 1: 직렬 가설 테스트 — qwen3 12연속 (#366~#377)

- **관측**: 5가지 가설(GOAWAY, TCP RST, Content-Length, chunked, CORS)을 하나씩 순차 테스트.
  각 가설마다 코드 수정 → 빌드 → 배포 → 테스트 요청 → 폴링 대기 → 결과 확인 사이클.
- **데이터**: 12건 × 평균 15분/건 = ~3시간. 이 중 10건이 STUCK_ON_THINKING으로 동일 결과.
- **영향**: 가설 3~5(Content-Length, chunked, CORS)는 모두 "SSE 파서 미동작"이라는 동일 증상.
  가설 3 실패 후 "HTTP/1.1 SSE 자체가 문제"라는 방향 전환이 가능했으나 가설 5까지 진행.
- **반론**: 각 가설은 독립적인 기술적 근거가 있었고, 체계적 배제법은 최종적으로 확실한 결론을 제공.
  그러나 **가설 3~5를 하나의 빌드에 통합 적용**했다면 2빌드 절약 가능.

### 패턴 2: page-load 즉시 차단 반복 (#350~#356)

- **관측**: deepseek, gemini에서 PAGE_LOAD_BLOCKED 발생. 사용자 피드백: "페이지 로드 즉시 차단은 서비스 제공 시 절대 하지 말아야 하는 동작".
- **데이터**: 4건 테스트에서 page-load 차단 확인. 근본 원인은 POST body 키워드 매칭이 아닌 URL/도메인 패턴 매칭에서의 오동작.
- **영향**: 사용자 신뢰도 저하, 테스트 사이클 낭비.
- **교훈**: **매 빌드 후 "페이지 정상 로드 확인" 테스트를 필수**로 수행해야 함.

## 긍정적 패턴

1. **체계적 가설 배제법**: GOAWAY → TCP RST → Content-Length → chunked → CORS → JSON 전환.
   최종적으로 "HTTP/1.1 SSE 주입 불가" 라는 확실한 결론 도출. 향후 유사 서비스에 재활용 가능.
2. **빠른 방향 전환**: 10회 SSE 실패 후 JSON 에러 접근으로 전환, 첫 시도에 성공 (#376).
3. **issue-h2mode-goaway-analysis.md 문서화**: 전체 가설 변천, 실험 결과, 결론을 체계적으로 기록.
   이 문서는 다른 HTTP/1.1 서비스 작업 시 귀중한 참조 자료.
4. **빌드 상한 준수 (이전 회고 개선안 지속)**: Phase3-B25 시리즈에서도 과다 빌드 없이 진행.
5. **사용자 피드백 즉시 반영**: page-load 즉시 차단 문제를 즉시 수정.

  실패 시 개별 원인을 서버 로그로 구분.
  ```
  apf-warning-impl Step 2 추가:
  - [ ] 유사 가설 2개 이상이면 통합 빌드 고려
  - [ ] 통합 불가 사유 명시 (예: 코드 경로 충돌)
  ```
- **대상**: `apf-warning-impl/SKILL.md`
- **기대**: 유사 가설 3건 → 1빌드. 빌드당 15분 절약 × 2 = 30분/서비스.

### [HIGH] 개선안 #4: HTTP/1.1 서비스 JSON 에러 전환 자동 판별

- **기대**: HTTP/1.1 SSE 서비스에서 10회 실패 방지. 즉시 JSON 전환.
- **우선순위**: HIGH
