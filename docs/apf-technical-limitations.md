## 1. WebSocket 서비스 — 경고 전달 방법 탐색

현재 APF는 HTTP/1.1 및 HTTP/2 응답 수준에서 템플릿을 주입한다.
WebSocket 기반 서비스에도 경고 문구를 전달하기 위해 다음 방법을 **모두 시도**한다.

| 서비스 | WS 엔드포인트 | 현재 상태 |
|--------|-------------|----------|
| character.ai | neo.character.ai (WS) | 도메인 차단 중, 경고 미표시 |
| poe.com | GraphQL over WS | 도메인 차단 중, 경고 미표시 |

### 시도할 방법 (우선순위순)

1. **HTTP Upgrade 응답 단계 인터셉트**
   - WebSocket 핸드셰이크는 HTTP/1.1 Upgrade 요청으로 시작됨
   - APF가 Upgrade 요청을 감지하면 101 대신 HTML 에러 페이지(403 + 경고 메시지)를 반환
   - 브라우저는 WebSocket 연결 실패 → 서비스 프론트엔드의 에러 핸들링이 동작
   - **장점**: 기존 HTTP 응답 메커니즘 활용, WS 프레임 파싱 불필요

2. **초기 페이지 로드 시 경고 페이지 주입**
   - 서비스의 HTML 페이지 로드 요청(GET /)을 인터셉트
   - 원본 페이지 대신 경고 메시지가 포함된 HTML 페이지를 반환
   - WebSocket 연결 시도 자체가 발생하지 않음
   - **장점**: 가장 확실한 경고 표시, 모든 서비스에 적용 가능
   - **고려사항**: 특정 path가 아닌 페이지 로드 요청 식별 필요 (Accept: text/html)

3. **WS 프레임 레벨 인젝션 (APF 기능 확장)**
   - APF에 WebSocket 프레임 파싱 + 주입 기능 추가
   - 민감 정보가 포함된 WS 메시지 감지 → 경고 메시지 프레임으로 응답
   - **장점**: 가장 정밀한 제어, 메시지 레벨 필터링
   - **고려사항**: C++ 코드 수정 필요, WS 프레임 형식(opcode, masking) 처리

4. **서비스별 REST API 병행 차단**
   - 일부 WS 서비스도 초기화/인증에 REST API를 사용
   - REST API 단계에서 차단 + 경고 반환
   - character.ai: 채팅 시작 시 REST로 캐릭터 정보 조회 → 이 단계에서 차단 가능
   - poe.com: GraphQL HTTP 폴백 엔드포인트 존재 가능 → 조사 필요

5. **DNS/리다이렉트 방식**
   - VT 모듈에서 해당 도메인에 대한 HTTPS 요청을 경고 페이지 서버로 리다이렉트
   - **고려사항**: 인증서 불일치 경고 발생 가능 → VT가 적합한 인증서 제공 필요

### 즉시 실행할 작업
- [ ] character.ai: REST API 엔드포인트 조사 (HAR 캡처)
- [ ] poe.com: GraphQL HTTP 폴백 존재 여부 확인
- [ ] 방법 1 (Upgrade 인터셉트) 프로토타입: APF에서 `Connection: Upgrade` 헤더 감지 시 경고 반환 가능한지 코드 분석
- [ ] 방법 2 (페이지 로드 인터셉트) 프로토타입: `Accept: text/html` 요청에 경고 HTML 페이지 반환 테스트

## 4. GOAWAY vs 스트리밍 전달
h2_mode=1 (GOAWAY)는 SSE/스트리밍 응답 전달을 차단. 해결: h2_mode=2 (keep-alive)로 전환.

## 공통 전략: 페이지 로드 인터셉트 (범용)
모든 서비스에 적용 가능한 범용 방법:
- 서비스 도메인에 대한 첫 번째 HTML 페이지 요청(`Accept: text/html`)을 감지
- 원본 페이지 대신 **경고 메시지가 포함된 커스텀 HTML 페이지**를 반환
- 이 방법은 API 형식, WS/SSE/GraphQL 등 프로토콜에 무관하게 동작
- **조건**: APF가 `Accept` 헤더를 분석하고 HTML 페이지 요청을 구별할 수 있어야 함
- **조사**: 현재 APF 코드에서 Accept 헤더 기반 분기 가능 여부 확인 필요
