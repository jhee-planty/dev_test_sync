WHY DEVTOOLS SHOWS 0 RESPONSE HEADERS:
  This is NOT proof that the engine emits 0 headers.
  When Chrome blocks a CORS preflight/credentialed-mode failure at the network layer,
  it surfaces the request as "(failed) CORS error" and the response-headers section
  reads "(0)" because Chrome refuses to expose them to the page (and to DevTools).
  v2's "0 headers" symptom was the SAME root cause (CORS), not an engine emission bug.

WHAT v3 ACTUALLY CHANGED (and didn't fix):
  v3 simplified Cache-Control / removed X-Accel-Buffering / added X-Content-Type-Options.
  But the missing piece is Access-Control-Allow-Credentials: true.
  The engine likely strips it (or never injects it) on the intercepted PII-block response,
  while the request rides on credentials: 'include' (cookies/session).
  Browser-side strict CORS check: ACAC must equal 'true' for credentialed fetches.
  Without it, Chrome wipes the entire response — body AND headers — from JS.
