ROOT CAUSE EVIDENCE
-------------------
The TRPCClientError shape has:  cause:undefined, data:undefined, meta:undefined, shape:undefined.
Yet the raw HTTP response DID contain "data":{"json":{...}} AND "meta":{"values":{...}}.
This strongly suggests the trpc client throws because either:
  (a) the tRPC envelope is missing required v11 keys (e.g. response is single-element array but client expects different shape), OR
  (b) the SuperJSON deserializer rejects the meta.values map because some field paths reference indexes the deserializer can't find (e.g. "messages.0.contentChunks.0._context":["undefined"] applied to value that doesn't actually exist in tree, or the meta describes "undefined" markers for fields that ARE present in payload).

Key suspect: meta.values lists "messages.0.contentChunks.0._context":["undefined"] but the actual payload has _context:null (NOT undefined). SuperJSON would try to set undefined where null exists → potential type-mismatch reject. Similarly "messages.0.agentId":["undefined"] applied to agentId:null.

This is HP-3 candidate: meta envelope describes serialization markers inconsistent with payload, causing SuperJSON deserialize to throw → wrapped as "Unknown error" by tRPC client → bubble never renders.
