Root cause hypothesis:
The h2_end_stream=1 (2-frame atomic, body + empty END_STREAM in same write batch)
appears to be triggering an H2 protocol error that bursts the whole connection,
not just the single stream. With h2_end_stream=2 (RST_STREAM) in #617, only the
specific stream was reset, and the client at least observed the headers + buffered
events before the RST arrived. With h2_end_stream=1, the cleaner-looking atomic
"body + empty END_STREAM frame" appears to violate something nghttp2 / Chrome
expects — possibly:
  (a) the empty DATA-with-END_STREAM frame is being rejected as malformed,
  (b) the stream is being closed while the SETTINGS exchange or HEADERS write
      hasn't completed and the H2 framer treats it as a protocol violation,
  (c) the parser fix flushed a different write order that put the END_STREAM
      frame BEFORE the HEADERS frame (or before the response body bytes were
      flushed to the wire), causing FRAME_SIZE_ERROR / PROTOCOL_ERROR on the
      receiving side.

The visible smoking gun is net::ERR_HTTP2_PROTOCOL_ERROR on the sibling cyoa
request — Chrome only reports this when nghttp2 (or h2 layer) actually parses
something invalid on the wire, NOT when the server resets a stream cleanly.

Recommendation:
1. Capture a wireshark / curl --http2-prior-knowledge -v trace of the stream
   exchange to see the exact frame ordering of the new atomic write.
2. Compare frame ordering between #617 (RST_STREAM, working) and #619 (atomic
   END_STREAM, broken).
3. Most likely fix: ensure HEADERS frame is fully flushed BEFORE the body+END_STREAM
   batch is queued; the new "atomic 2-frame batch" may be racing the HEADERS write.
